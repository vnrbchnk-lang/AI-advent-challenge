# Week 04 — MCP: summary

## Суть лекции

Model Context Protocol (MCP) — открытый стандарт, который связывает AI-приложения с внешними
инструментами и данными. LLM сама по себе не знает актуального состояния корпоративных систем,
таск-трекеров, репозиториев, календарей, внутренних файлов и CI/CD. MCP вводит общий слой: host с
LLM использует MCP client, server публикует доступные tools, а обработчик переводит tool call в
конкретный API request, CLI-команду или вызов внутренней функции.

MCP не является framework и не заменяет API. SDK на Kotlin, Python или TypeScript лишь реализуют
протокол. API остаётся интерфейсом конкретного backend, а MCP server адаптирует его для AI hosts.
Благодаря этому одну интеграцию можно подключать к нескольким совместимым клиентам, не реализуя
отдельную связь для каждой пары «модель — сервис».

## Ключевые тезисы

- MCP особенно полезен для корпоративных систем, баз знаний, репозиториев, DevOps, календарей,
  почты и сложных приложений вроде CRM, 1C или SAP.
- Tool — отдельная операция с именем, description и input schema. Server может предоставить только
  безопасное подмножество возможностей, например чтение без записи.
- JSON-RPC 2.0 задаёт форму сообщений. stdio transport применяется для локального subprocess, а
  remote transport — для сетевого сервера. Формат сообщений и способ доставки — разные уровни.
- Remote MCP Server требует размещения, реализации протокола, доступа к backend, authentication,
  TLS, publication и подключения к AI host.
- В Kotlin-примере MCP server создаётся через SDK и stdio transport. Tool `get_forecast` принимает
  latitude и longitude, валидирует их, вызывает Ktor HTTP client для weather API и возвращает
  `CallToolResult` с `TextContent`.
- Главный mapping выполняет handler: формальный tool call превращается в конкретный HTTP request.
- MCP упрощает повторное использование интеграций, но добавляет server infrastructure, security
  risks и token overhead.

Безопасность нельзя сводить к наличию schema. Внешний текст может содержать prompt injection, а
неосторожный handler — передать вредоносное значение в shell или критический API. Практическая
защита включает least privilege, read-only режим, строгую validation, allowlists, sandbox,
timeouts, audit log, отдельные credentials и подтверждение destructive actions.

## Практические выводы

Начинать внедрение стоит с одного ограниченного read-only use case: поиска по базе знаний,
получения задач или состояния pipeline. Для каждого tool нужно определить владельца, scope,
validation, error model и правила logging. После этого можно измерять ценность: время выполнения,
долю успешных операций, token cost и количество заблокированных опасных запросов.

MCP полезен не только как техническая интеграция, но и как организационный механизм. Команда может
публиковать внутренние tools, подключать корпоративный AI, обучать пользователей и формировать
центр компетенций. При этом естественный язык не отменяет бизнес-правила: агент должен соблюдать
те же ограничения доступа, что и обычные клиенты системы.

Для AI Advent Challenge важно сравнивать подходы воспроизводимо. Одинаковую задачу следует
запускать через MCP и Skill + CLI, фиксируя schemas, число sampling calls, tool results, итоговые
prompt tokens и удобство эксплуатации. Day 16 реализует первый клиентский шаг: подключение к
публичному DeepWiki MCP по Streamable HTTP и discovery через `tools/list` без вызова tools.

## Риски и ограничения

Token cost появляется прежде всего в sampling calls к LLM. В prompt входят system instructions,
user input, история, tool schemas и результаты предыдущих операций. При длинной цепочке каждый
следующий шаг может повторно передавать растущий контекст.

Основные источники overhead:

- idle overhead — descriptions и schemas присутствуют, хотя конкретные tools не используются;
- batching overhead — множество отдельных действий порождает повторные calls и накопленную историю;
- schema weight — большое количество подробных tools увеличивает input.

Пример лекции использует system prompt около 1500 токенов, user input около 200, шесть schemas
суммарно около 5400, tool result около 3500 и response около 600. Это иллюстрация масштаба, а не
точный benchmark: числа частично уточняются по ходу объяснения. Универсального квадратичного закона
для MCP нет; результат зависит от host, caching, compacting, модели и сборки prompt.

Дополнительное ограничение — MCP server не всегда оправдан. Если один локальный агент работает с
простым личным сервисом, прямой API или существующий CLI может потребовать меньше кода и контекста.

## MCP vs Skill + CLI в одном абзаце

MCP лучше подходит для удалённых серверов, публичных tools, нескольких AI hosts, централизованных
credentials и формальных schemas. Skill + CLI часто выгоднее в локальной среде, где готовая
утилита уже установлена, инструкции загружаются on demand, несколько операций можно batch-ить, а
idle overhead равен нулю. Поэтому тезис `MCP deprecated` не означает отмену технологии: он
подчёркивает, что для части локальных и CLI-задач MCP может быть дороже и сложнее. Практический
выбор определяется topology, security model, повторным использованием и измеренным token flow.
