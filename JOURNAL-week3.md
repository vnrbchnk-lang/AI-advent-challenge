### День 11 — Модель памяти ассистента (memory layers)
- Статус: код готов, ревью пройдено; живой прогон — за пользователем (правило 11, экономия токенов)
- Файлы: `week3/assistant/memory.py`, `week3/assistant/cli11.py`, `week3/pyproject.toml`
- Задание: описать и реализовать модель памяти ассистента, минимум 3 типа — краткосрочная (диалог), рабочая (данные текущей задачи), долговременная (профиль/решения/знания); хранить отдельно; явно выбирать, что и куда сохраняется; проверить, какие данные попадают в каждый слой и как это влияет на ответы.
- Архитектура: `MemoryLayers` (memory.py) — три слоя, каждый в **отдельном файле** `week3/assistant/store/`: `short_term.json` (список реплик), `working.json` (KV-данные задачи), `long_term.json` (KV-профиль/решения). «Хранятся отдельно» — буквально три файла (нагляднее для видео). `build_messages()` — prompt builder: `[system] + [system: долговременная] + [system: рабочая] + последние N реплик`. Слои подмешиваются в запрос **явными флагами** `use_long_term`/`use_working` — можно включать/выключать, чтобы показать влияние на ответ.
- Явный выбор «что куда» — через слэш-команды CLI (а не авто-роутер): `/remember-forever название = содержимое` → долговременная; `/remember-now название = содержимое` → рабочая; обычное сообщение (без слэша) → краткосрочная (диалог). Память — KV-заметки (название=содержимое); без «=» название генерится само.
- Набор команд намеренно минимальный (6, после упрощения по просьбе пользователя — раньше было 13, «невозможно разобраться»): `/remember-forever` (→ долговременная), `/remember-now` (→ рабочая/задача), `/show-memory`, `/demo`, `/reset`, `/help`. Убраны `/forget`, `/newtask`, `/clear`, `/prompt`, `/longterm on|off`, `/working on|off` — гранулярная чистка и тумблеры слоёв перегружали, а влияние памяти и так показывает `/demo`. Методы чистки остались в `memory.py` (пригодятся в следующих днях недели), просто не выведены в CLI.
- Меню команд: при наборе `/` всплывает цветной список (`prompt_toolkit`, своя `Style`), навигация стрелками ↑↓, выбор Enter (биндинг: Enter при выбранной подсказке фиксирует команду и НЕ отправляет строку — `completion_is_selected`). Подмешивание в промпт сейчас автоматическое для всех слоёв (тумблеры убраны; явный — только выбор слоя при записи, что и требует ТЗ). Антипаттерн «всё в каждый prompt» осознан, выборочное подмешивание — задача prompt builder ближе к дню 13.
- Проверка «что в каждом слое»: команда `/show-memory` — таблица из 3 слоёв с колонкой «что хранит».
- Проверка «как влияет на ответы»: команда **`/demo`** внутри CLI — A/B на одном вопросе. Берёт **реальную долговременную память пользователя** (если пуста — образец `DEMO_PROFILE`); вопрос можно задать свой: `/demo <вопрос>` (дефолт — про сервис авторизации). Слева запрос БЕЗ памяти → обобщённый ответ; справа С памятью → ответ учитывает её факты. Демо гоняет на временных сообщениях, живую память не трогает.
- Модель: **`gpt-4.1`** (смена с gpt-5.2 ради экономии, решение пользователя — выход 2062₽/1М против 4245₽ у gpt-5.2, вдвое дешевле; цена сверена на https://proxyapi.ru/pricing/list). Учёт токенов и цены (₽, прайс ProxyAPI) в подвале каждого хода.
- CLI: красивый интерфейс через `rich` (правило 12) — панели, цветные таблицы слоёв, спиннер «думает», баннер на старте. Слэш-команды как в Claude Code: ввод через `prompt_toolkit`, при наборе `/` всплывает список команд с описанием (автодополнение). `/help` — подробная справка с объяснением KV-памяти.
- Глобальная команда: изначально **`agent11`** → `assistant.cli11:main` (день-11-онли). Позже (дни 12–14) `agent11` переключён на полный агент `assistant.cli:main`, охватывающий дни 11–14; `cli11.py` остался файлом-артефактом без скрипта.
- Сверх ТЗ: глобальная команда + rich-интерфейс; флаги подмешивания слоёв и команда `prompt` (видно сборку запроса); персистентность всех трёх слоёв между запусками; раздельный учёт токенов/цены. Заявка на неделю: это первый кирпич растущего stateful-агента (дальше — профиль/state machine/инварианты).
- Запуск (пользователь): `pip install -e week3` → `agent11`, демо — команда `demo` внутри CLI. `store/*.json` в .gitignore.

### Дни 12–14 — Собранный stateful-агент (профиль + state machine + инварианты)
- Статус: код готов, ревью+смоук детерминированной логики пройдены; живой прогон LLM — за пользователем (правило 11)
- Решение: по явной просьбе пользователя написан сразу **полный агент со всеми способностями дней 11→14** в одном пакете (а не по одному дню). Каждый день показывается отдельной демо-командой по нарастанию: `/demo1` (память), `/demo2` (профиль), `/demo3` (state machine), `/demo4` (инварианты).
- Демо построены по принципу **«с твоими настоящими данными vs БЕЗ них»** (просьба пользователя): два запроса на один вопрос — слева без слоя, справа с реальными данными пользователя (память/профиль/инварианты), берётся образец только если данных ещё нет. Сценарий видео: пользователь заполняет данные (`/interview`, `/remember-forever`, `/invariant-add`, `/task`), затем жмёт демо — видно вклад именно этого слоя. Демо принимают свой вопрос: `/demoN <вопрос>`.
- В интерфейсе агента **убраны любые упоминания номеров дней** (просьба пользователя) — фичи названы по сути (память/профиль/состояние/инварианты).
- Полный конвейер стадий показан в UI как фича: `planning → execution → validation → done` (с подсветкой текущей) в `/status`, при `/task` и в баннере.
- Единая точка входа: команда **`agent11`** теперь указывает на полный агент (`assistant.cli:main`) — она же охватывает день 11 (демо памяти `/demo1`). Старый день-11-онли `cli11.py` остаётся файлом-артефактом, но без отдельного скрипта.
- `/interview` — стартовое интервью (день 12): агент задаёт вопросы (стиль/формат/уровень/роль) и пишет профиль; альтернатива ручному `/profile-set`. Без LLM-вызовов.
- Файлы: `week3/assistant/profile.py`, `state.py`, `invariants.py`, `validator.py`, `prompt_builder.py`, `cli.py`; правка `pyproject.toml` (скрипт `agent`).

**День 12 — Персонализация (`profile.py`)**
- Задание: создать профиль пользователя; описать предпочтения (стиль/формат/ограничения); подключить профиль к каждому запросу; проверить ответы для разных профилей и что учитывается автоматически.
- Сделано: `Profile` — отдельный файл `store/profile.json`, поля `стиль/формат/уровень/роль`. Заполнение двумя путями: вручную `/profile-set поле = значение` или интервью `/interview` (агент задаёт вопросы по полям). Подключение к каждому запросу — через `prompt_builder.build_messages` (профиль идёт отдельным system-блоком). Проверка: `/demo2` гоняет один вопрос дважды — БЕЗ профиля и С твоим реальным профилем — бок о бок; в вопросе про стиль/формат не сказано, агент берёт из профиля сам.
- Развязка профиль/инварианты (решение пользователя): профиль = мягкое (стиль/формат/кто-юзер), инварианты = твёрдое (стек/архитектура/бизнес, отказ при нарушении). «Запреты» из дня 11 концептуально переезжают в инварианты.

**День 13 — Task state machine (`state.py`)**
- Задание: состояние задачи как конечный автомат (этап / текущий шаг / ожидаемое действие); пример planning→execution→validation→done; проверить паузу на любом этапе и продолжение без повторных объяснений.
- Сделано: `TaskStore` (мультизадачный, переписан из одиночного `TaskState` по просьбе пользователя) — стадии `STAGES`, `TRANSITIONS` (allowed transitions), `EXPECTED` (ожидаемое действие), словарь задач `{имя: {stage, step, plan, results, status}}` + указатель `current` в `store/state.json`. Статусы: `active` (в работе) / `paused` (пауза) / `done` (готово).
- UX переделан (просьба пользователя): **одна команда `/task <имя>`** создаёт новую или продолжает существующую задачу и **открывает её отдельное рабочее пространство** (`run_workspace`) — вложенный prompt-loop со своим приглашением `имя ›`, заголовком-карточкой и навигацией. Внутри: `/plan <текст>` (утвердить план), `/next` / `/back` (вперёд/назад по стадиям, легальность проверяет КОД), `/pause` (приостановить и выйти), `/delete` (удалить), `/exit` (выйти, задача остаётся), а просто текст = вопрос агенту в контексте этой задачи (стадия подмешивается в промпт).
- **Список всех задач со статусами виден всегда** в нижней панели (`bottom_toolbar` prompt_toolkit) — и в общем чате, и внутри рабочего пространства; плюс таблица задач в `/status`. Полный конвейер `planning → execution → validation → done` показан с подсветкой текущей стадии.
- **Легальность перехода проверяет КОД** (`transition()`), не промпт: перепрыгивание этапа отклоняется, execution без плана отклоняется, done терминальна. Пауза = стадия/план персистятся в файл на каждом шаге; продолжение без повторных объяснений = `/task <имя>` грузит стадию/план/результаты. Несколько задач живут параллельно, переключение через `/task <другое имя>`. Состояние текущей задачи подмешивается в каждый запрос отдельным system-блоком.
- `/demo3` — A/B на один запрос «дай сразу код»: БЕЗ состояния агент прыгает в код, С состоянием (стадия planning) держит процесс и требует сперва план. Плюс карта автомата (с подсветкой текущей стадии), детерминированный отказ кода на нелегальном переходе (× planning→done) и блок про паузу/резюм. Берёт реальную стадию задачи, если она активна.

**День 14 — Инварианты и ограничения (`invariants.py` + `validator.py`)**
- Задание: инварианты, которые агент не имеет права нарушать; хранить отдельно от диалога; явно учитывать в рассуждениях; отказываться от решений, нарушающих их; проверить конфликт запроса и инварианта и как объясняется отказ.
- Сделано: `Invariants` — отдельный файл `store/invariants.json`, каждый инвариант = `{rule, forbid[]}`. Хранятся отдельно от диалога (свой файл). Учёт в рассуждениях — блок инвариантов идёт в system-промпт первым (самый приоритетный). Две линии защиты (решение пользователя — «код-линтер + LLM-критик», конспект требует «и промптом, и кодом»):
  1. **код-линтер** `Invariants.lint()` — детерминированный, ищет стоп-слова в ответе; работает на каждом обычном ходе (предупреждение в подвале).
  2. **LLM-критик** `validator.critic_check()` на **`gpt-4o-mini`** (дешёвый второй агент) — семантическая проверка ответа против инвариантов, возвращает JSON `{violated, which, why}`.
- Доп. агент (решение пользователя — ровно один, без роя): критик-валидатор = он же **стадия `validation`** дня 13. При `/go validation` агент гоняет критика на последнем ответе → дни 13 и 14 сходятся в одной точке.
- `/demo4` — A/B на один запрос, конфликтующий с инвариантами: БЕЗ инвариантов агент выполняет запрос, С твоими инвариантами отказывается и объясняет. Затем по ответу БЕЗ инвариантов прогоняются (2) код-линтер (стоп-слова) и (3) LLM-критик (семантика) — оба ловят нарушение. Инварианты реальные пользователя (образец, если пусто).

**Общее (сверх ТЗ / архитектура)**
- `prompt_builder.build_messages` — сборка запроса из слоёв в порядке приоритета: base → инварианты → профиль → состояние задачи → долговременная → рабочая → окно диалога. Реализует «общую архитектуру» из конспекта (п.135): profile storage / task state storage / invariant checker / response validator / prompt builder / LLM engine.
- `/status` — единая панель всего состояния (3 слоя памяти + профиль + инварианты + стадия задачи) для наглядности на видео.
- Команды общего чата (13): `/status`, `/task`, `/interview`, `/remember-forever`, `/remember-now`, `/profile-set`, `/invariant-add`, `/demo1`, `/demo2`, `/demo3`, `/demo4`, `/reset`, `/help`. Команды рабочего пространства задачи (8): `/plan`, `/next`, `/back`, `/status`, `/pause`, `/delete`, `/help`, `/exit`.
- Модели: основной `gpt-4.1`, критик `gpt-4o-mini`. CLI — rich + prompt_toolkit, сталь-серый, без эмодзи (правила 10/12).
- Запуск (пользователь): `pip install -e week3` → `agent11`; демо по дням внутри CLI: `/demo1` (память) `/demo2` (профиль) `/demo3` (state machine) `/demo4` (инварианты). `store/*.json` в .gitignore.
