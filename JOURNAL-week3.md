- Архитектура: `MemoryLayers` (memory.py) — три слоя, каждый в **отдельном файле** `week3/assistant/store/`: `short_term.json` (реплики), `working.json` (KV задачи), `long_term.json` (KV профиль/решения). «Отдельно» = буквально три файла (нагляднее для видео). `build_messages()` — prompt builder: `[system] + [system: долговременная] + [system: рабочая] + последние N реплик`. ### Дни 12–14 — Собранный stateful-агент (профиль + state machine + инварианты)
- Статус: код готов, ревью+смоук детерминированной логики пройдены; живой прогон LLM — за пользователем (правило 11)
- Решение: по просьбе написан сразу **полный агент со всеми способностями дней 11→14** в одном пакете (не по дню). Каждый день — отдельная демо-команда по нарастанию: `/demo1` (память), `/demo2` (профиль), `/demo3` (state machine), `/demo4` (инварианты).
- Демо по принципу **«с твоими настоящими данными vs БЕЗ них»** (просьба): два запроса на вопрос — слева без слоя, справа с реальными данными (память/профиль/инварианты), образец только если данных нет. Сценарий видео: пользователь заполняет (`/interview`, `/remember-forever`, `/invariant-add`, `/task`), жмёт демо — виден вклад слоя. Демо принимают свой вопрос: `/demoN <вопрос>`.
- В интерфейсе **убраны упоминания номеров дней** (просьба) — фичи названы по сути (память/профиль/состояние/инварианты).
- Полный конвейер стадий как фича: `planning → execution → validation → done` (с подсветкой текущей) в `/status`, при `/task`, в баннере.
- Единая точка входа: **`agent11`** теперь → полный агент (`assistant.cli:main`), охватывает и день 11 (`/demo1`). Старый `cli11.py` — артефакт без скрипта.
- `/interview` — стартовое интервью (день 12): агент спрашивает (стиль/формат/уровень/роль) и пишет профиль; альтернатива ручному `/profile-set`. Без LLM.
- Файлы: `week3/assistant/profile.py`, `state.py`, `invariants.py`, `validator.py`, `prompt_builder.py`, `cli.py`; правка `pyproject.toml` (скрипт `agent`).

- Задание: создать профиль; описать предпочтения (стиль/формат/ограничения); подключить к каждому запросу; проверить ответы для разных профилей и что учитывается авто.
- Сделано: `Profile` — отдельный файл `store/profile.json`, поля `стиль/формат/уровень/роль`. Заполнение двумя путями: вручную `/profile-set поле = значение` или интервью `/interview`. Подключение к запросу — через `prompt_builder.build_messages` (профиль отдельным system-блоком). Проверка: `/demo2` гоняет вопрос дважды — БЕЗ профиля и С реальным — бок о бок; стиль/формат в вопросе не сказан, агент берёт из профиля сам.
- Развязка профиль/инварианты (решение): профиль = мягкое (стиль/формат/кто-юзер), инварианты = твёрдое (стек/архитектура/бизнес, отказ при нарушении). «Запреты» дня 11 концептуально → инварианты.

**День 13 — Task state machine (`state.py`)**
- Задание: состояние задачи как конечный автомат (этап / шаг / ожидаемое действие); пример planning→execution→validation→done; проверить паузу на любом этапе и продолжение без повторных объяснений.
- Сделано: `TaskStore` (мультизадачный, переписан из одиночного `TaskState` по просьбе) — стадии `STAGES`, `TRANSITIONS` (allowed transitions), `EXPECTED` (ожидаемое действие), словарь `{имя: {stage, step, plan, results, status}}` + указатель `current` в `store/state.json`. Статусы: `active` / `paused` / `done`.
- UX переделан (просьба): **одна команда `/task <имя>`** создаёт или продолжает задачу и **открывает её рабочее пространство** (`run_workspace`) — вложенный prompt-loop со своим приглашением `имя ›`, карточкой-заголовком, навигацией. Внутри: `/plan <текст>` (утвердить план), `/next` / `/back` (легальность проверяет КОД), `/pause` (приостановить и выйти), `/delete`, `/exit` (выйти, задача остаётся), текст = вопрос агенту в контексте задачи (стадия в промпте).
- **Список всех задач со статусами виден всегда** в нижней панели (`bottom_toolbar`) — в чате и внутри рабочего пространства; плюс таблица в `/status`. Конвейер `planning → execution → validation → done` с подсветкой текущей.
- **Легальность перехода проверяет КОД** (`transition()`), не промпт: перепрыг отклоняется, execution без плана отклоняется, done терминальна. Пауза = стадия/план персистятся на каждом шаге; продолжение без повторов = `/task <имя>` грузит стадию/план/результаты. Несколько задач параллельно, переключение `/task <другое имя>`. Состояние текущей задачи — отдельным system-блоком в каждом запросе.
- `/demo3` — A/B на «дай сразу код»: БЕЗ состояния агент прыгает в код, С состоянием (planning) держит процесс и требует план. Плюс карта автомата (подсветка стадии), детерминированный отказ кода на нелегальном переходе (× planning→done), блок про паузу/резюм. Берёт реальную стадию, если активна.

**День 14 — Инварианты и ограничения (`invariants.py` + `validator.py`)**
- Задание: инварианты, что агент не вправе нарушать; хранить отдельно от диалога; явно учитывать в рассуждениях; отказываться от нарушающих решений; проверить конфликт запроса и инварианта и как объясняется отказ.
- Сделано: `Invariants` — отдельный файл `store/invariants.json`, каждый = `{rule, forbid[]}`. Хранятся отдельно (свой файл). Учёт — блок инвариантов в system-промпт первым (приоритетный). Две линии защиты (решение — «код-линтер + LLM-критик», конспект требует «и промптом, и кодом»):
  1. **код-линтер** `Invariants.lint()` — детерминированный, ищет стоп-слова в ответе; на каждом обычном ходе (предупреждение в подвале).
  2. **LLM-критик** `validator.critic_check()` на **`gpt-4o-mini`** (дешёвый второй агент) — семантическая проверка против инвариантов, возвращает JSON `{violated, which, why}`.
- Доп. агент (решение — ровно один, без роя): критик-валидатор = **стадия `validation`** дня 13. При `/go validation` агент гоняет критика на последнем ответе → дни 13 и 14 сходятся.
- `/demo4` — A/B на запрос, конфликтующий с инвариантами: БЕЗ инвариантов агент выполняет, С твоими отказывается и объясняет. Затем по ответу БЕЗ прогоняются (2) код-линтер и (3) LLM-критик — оба ловят нарушение. Инварианты реальные (образец, если пусто).

### День 15 — Контролируемые переходы состояний + рой агентов
- Статус: код готов, ревью+смоук детерминированной логики переходов пройдены; живой прогон LLM (рой) — за пользователем (правило 11)
- Задание (дословно): явные переходы между состояниями — допустимые состояния; разрешённые переходы; ассистент не может «перепрыгнуть» этап (нельзя реализация до плана; нельзя финал без валидации). Проверить: попытки недопустимого перехода, реакцию, продолжение после паузы.
- Главное: **ядро ТЗ уже готово с дня 13** — `STAGES`, `TRANSITIONS`, код-гейт `TaskStore.transition()` отклоняет нелегальные переходы, execution заблокирован без плана, done терминальна. Конспект подтверждает: «allowed transitions добавляют детерминизм», «для жёстких запретов нужен код — текстовые правила теряются после summary/compacting», антипаттерн «не валидировать переходы state machine».
- Добавил день 15:
  1. **`/demo5`** (`run_demo_transitions`) — детерминированная демонстрация ТЗ, **без LLM** (экономия): таблица состояний и переходов; попытки перепрыгнуть (planning→done через 2 этапа, planning→validation, execution без плана, done→execution) — каждая отклонена кодом с причиной, состояние не меняется; легальный путь planning→execution→validation→done проходит; блок «пауза → статус/стадия/план/результаты в `store/state.json`; продолжение `/task <имя>` поднимает состояние — без повторов». Закрывает все 3 проверки ТЗ.
  2. **Рой агентов на каждой стадии** (`swarm.py`, **сверх дословного ТЗ** — по настоянию оргкомитета челленджа: «обмазать валидациями и подключить рой… 5 агентов на этап, обмен мнениями через оркестратора, он должен возразить»). По доработанной просьбе: **на каждой рабочей стадии трудится свой рой**, человек только разрешает переход. 5 агентов-ролей (под стадию: planning — Архитектор/Безопасность/Производительность/Прагматик/Критик; execution — Бэкенд/Данные/Тесты/Интеграции/Критик кода; validation — Инварианты/Безопасность/Граничные случаи/Производительность/Критик) на дешёвом **`gpt-4o-mini`**; **оркестратор** на **`gpt-4.1`** знает запрос + профиль + инварианты + утверждённый план + результаты прошлых стадий, сводит мнения и **возражает** при нарушении инварианта (формат `ВОЗРАЖЕНИЯ:` / `ИТОГ:`). Стадия `done` — без роя, финальная сводка (`show_done_report`, 0 токенов).
  3. **Рой запускается авто при входе на стадию** (решение): создание задачи → рой planning; каждый `/next` → рой новой стадии. `/council` — ручной перезапуск роя текущей стадии.
- **Человек — единственный воротчик; LLM не может сменить стадию** (главное ТЗ дня 15): переход выполняет только код по `/next`/`/back`. У LLM нет инструмента перехода — только текст. Усилено: в блоке состояния (`state.as_prompt`) инструкция «менять стадию ты не можешь, это делает пользователь командой /next; просят перейти/пропустить — откажись и объясни». Тест: пользователь просит «перейди на следующую стадию» → агент отказывается, стадия не меняется; двигает только `/next`.
- **Связка роя со state machine**: итог роя planning → утверждённый план → код только после этого разрешает `/next` в execution; итоги execution/validation → результаты задачи → контекст следующего роя. Рой встроен в контролируемый жизненный цикл, не сбоку.
- **Распределение токенов** (просьба «грамотно распределяй»): 5 членов роя — дешёвый `gpt-4o-mini`, дорогой `gpt-4.1` один раз на оркестратора стадии; `done` и `/demo5` — 0 токенов; рой работает в живой сессии при навигации (создание задачи / `/next`), не агентом Claude Code (правило 11). Подвал роя показывает раздельный учёт токенов роя и оркестратора (+ ₽ оркестратора; цену `gpt-4o-mini` не выдумывал — только токены). Авто-режим выбран осознанно (≈3 стадии × 6 вызовов на задачу).
