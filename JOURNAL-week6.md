Пакет `week6/local/`, команда `agent26` (`pip install -e week6`). Локальная LLM — Ollama
(`qwen2.5:7b-instruct`, Q4; для дня 29 ещё `qwen2.5:7b-instruct-q8_0`). CLI — rich + prompt_toolkit,

### День 26 — Запуск локальной LLM
- Статус: готово (ждёт живого прогона на видео)
- Файлы: `week6/local/localllm.py`, `week6/local/cli.py`
- Задание: установить/запустить локальную LLM; проверить локальный запуск, доступ по CLI/HTTP,
  ответ на простой запрос; сделать минимум 3 запроса разной сложности.
- Что сделано: Ollama + `qwen2.5:7b-instruct`. `/models` — список локальных моделей через HTTP
  `GET /api/tags` (доказательство «локально»). `/ask <текст>` — один запрос + статистика скорости.
  `/demo26` — 3 запроса разной сложности: простой факт → задача на рассуждение (пошагово) →
  генерация кода на Python; каждый с замером tok/s и латентности.
- Сверх ТЗ: замер скорости (tok/s, время загрузки) в каждом ответе.

### День 28 — Локальная LLM + RAG
- Статус: готово (локальную часть агент проверил; `/compare28` облачная сторона — за пользователем на видео)
- Файлы: `week6/local/ragbridge.py`, `week6/local/cli.py`
- Задание: подключить локальную LLM к RAG-пайплайну; индекс из недели, retrieval локально, генерация
  локальной моделью; сравнить с облаком (качество/скорость/стабильность).
- Что сделано: `ragbridge.py` импортирует пакет `rag` из недели 5 (через sys.path) — ретрив там уже
  полностью локальный (эмбеддинги bge-m3 + numpy-индекс). Генерацию подменил облако→локаль.
  `/localrag <q>` — весь RAG на машине (ретрив + генерация) + источники + тайминг. `/compare28 <q>`
  (`/demo28`) — тот же вопрос локальная vs облачная (`gpt-4.1`, ProxyAPI) рядом, с таймингами и
  подсказкой что сравнивать.
- Сверх ТЗ: переиспользование готового индекса недели 5 без пересборки.

### День 29 — Оптимизация локальной LLM
- Статус: готово (ждёт живого прогона на видео; нужна модель q8_0)
- Файлы: `week6/local/cli.py` (`_optimize`, `demo29`), `week6/local/localllm.py`
- Задание: оптимизировать модель под задачу — параметры (temperature/max tokens/context window),
  квантование, prompt-шаблон; сравнить качество до/после, скорость, ресурсы.
- Что сделано: `/optimize <q>` / `/demo29`, две части. (1) Параметры+промпт до/после на кейсе QA по
  базе (Q4): «до» дефолт (temp 0.3, num_ctx по умолчанию 2048 → контекст режется) → «после»
  (temp 0.1, num_ctx 4096, num_predict 220, заточенный prompt-шаблон). Параметры «после» меняются
  вручную командой `/params`. (2) Квантование Q4_K_M vs Q8_0 на коротком запросе. Таблицы скорости
  (секунды, tok/s, токены) + футпринт `GET /api/ps`.
- Ограничение железа (важно): Q8_0 (8GB) на 15GB RAM/CPU нестабилен при большом RAG-контексте —
  OOM. Поэтому: (а) квант-сравнение на коротком промпте; (б) `switch_to` освобождает ВСЕ модели
  (вкл. bge-m3) перед загрузкой Q8; (в) каждый квант-прогон в отдельном процессе (`_isolated_ask`,
  тайм-аут) — если Q8 не уместился, показывается панель «ресурсов не хватило», CLI не падает. Это и
  есть честный вывод по ресурсам (ТЗ: «квантование если доступно»).
- Сверх ТЗ: ручные параметры `/params`, изоляция прогонов в процессах, футпринт памяти, сводные
  таблицы скорости.
