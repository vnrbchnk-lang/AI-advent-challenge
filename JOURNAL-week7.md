# Журнал — Неделя 7 (production-ready AI-системы), дни 31–35

Пакет `week7/devassist/`, команда `agent31` (`pip install -e week7`). Один растущий пакет,
дни — команды одного CLI. Каркас взят из конспекта недели (`07_7_неделя_6_поток_v1_summary.md`):
агент как мини-ОС, Tool Registry + Tool Executor, субагенты (fan-out), AI-пайплайн на PR,
error handling с retry/fallback, метрики latency/cost/quality, human-in-the-loop на опасных операциях.

Кодовая база проекта — **«Личная культура»** (`C:\Alaba`, хакатон, Flutter + NestJS + Postgres/Prisma).
Репозиторий чужой (GitVerse `studentlabs/personal_culture_mobile`), в ADVENT **не копируется и не
коммитится**: инструменты читают его напрямую по `ALABA_ROOT`, индексы и песочница в `.gitignore`.

## Решения недели (зафиксировано 2026-07-19)

- **Не копировать чужой репозиторий, а читать по пути.** Копия ломает ТЗ дня 31 («получение
  *текущей* git-ветки» — в копии ветка навсегда `main`) и дня 32 (нужен реальный diff), дрейфует от
  оригинала и не снижает риск утечки: утечка происходит на вызове LLM, а не от расположения папки.
  Вместо копии — `ALABA_ROOT`, deny-list при индексации и скан на секреты.
- **Запись файлов — только в песочницу.** День 34 требует изменять файлы. Живой репозиторий
  защищён кодом (`WRITABLE_PROJECTS = {"sandbox"}`), песочница = `git clone C:\Alaba` в
  `week7/sandbox/` (оригинал не трогается вообще, git настоящий, diff настоящий, воспроизводимо).
- **Пайплайн дня 32 живёт в ADVENT.** Alaba на GitVerse, GitHub Action там не поднимется, а трогать
  чужой репозиторий CI-конфигами не стали. Action ревьюит PR в `AI-advent-challenge`, RAG-корпус —
  этот же репозиторий (CLAUDE.md с 12 правилами + журналы + код недель 1–7).
- **Эмбеддинги — облако.** `text-embedding-3-small` через ProxyAPI (эндпоинт проверен живым
  вызовом: 1536-dim, 200 OK; цена 5.16 ₽/1М). Неделя 5 брала локальный `bge-m3`, но модели Ollama
  были удалены после недели 6, а корпус тут в 5 раз больше — облако быстрее (967 чанков за 31 с).
  Генерация — `gpt-4.1`, вспомогательное (реранк) — `gpt-4o-mini`.
- **Порог косинуса 0.35** подобран по замерам на этом корпусе: вопросы по делу 0.47–0.57,
  оффтоп («как приготовить борщ») 0.20–0.28.
- **Три индекса:** `alaba` (967 чанков / 264 файла: доки, схемы Prisma и JSON, dart+ts),
  `advent` (236 / 66), `support` (78 / 5: FAQ + описание продукта).

## Устройство пакета

| Файл | Роль |
|---|---|
| `config.py` | проекты и их источники, deny-list, паттерны секретов, цены моделей |
| `corpus.py` / `chunking.py` / `index.py` / `retrieval.py` | RAG: сбор → чанкинг (md по заголовкам, код по окнам строк, схемы целиком) → npz+json индекс → поиск с порогом |
| `embedder.py` / `llm.py` | ProxyAPI: эмбеддинги и chat с retry ×3 и fallback на `gpt-4o-mini` |
| `metrics.py` | latency P50/P95, токены, стоимость в рублях по каждому этапу → JSONL, команда `/metrics` |
| `tools_project.py` / `tools_tickets.py` | реализация инструментов (git, файлы, поиск; тикеты) |
| `servers/project_app.py` / `servers/tickets_app.py` | два MCP-сервера (FastMCP, stdio) поверх них |
| `mcp_bridge.py` | фоновый asyncio-поток держит обе stdio-сессии открытыми, вызовы из CLI через `run_coroutine_threadsafe` |
| `registry.py` / `executor.py` | Tool Registry (MCP + локальные, схемы для function-calling) и Executor с гейтом опасных операций |
| `subagents.py` | fan-out делегирование (день 32) |
| `qa.py` / `review.py` / `support.py` / `agentloop.py` / `status.py` | дни 31 / 32 / 33 / 34 / 35 |
| `cli.py` | rich + prompt_toolkit, navy `#34568b`, без эмодзи |

### День 31 — Ассистент разработчика
- Статус: готово (ждёт живого прогона на видео)
- Файлы: `week7/devassist/qa.py`, `assistant.py`, `servers/project_app.py`, `cli.py` (`/help`, `/demo31`)
- Задание: RAG по документации проекта (README, docs, схемы данных/API); через MCP подключить
  ассистента к проекту (минимум — текущая git-ветка, по желанию файлы и diff); команда `/help`,
  которая отвечает на вопросы о проекте, используя документацию и контекст проекта.
- Что сделано: индекс `alaba` собран из README, `AGENTS.md`, всех 25 файлов `.memory-bank/`,
  `server/prisma/schema.prisma`, ~30 контрактов `shared/generated/*.schema.json`,
  `content/*.schema.json` и исходников (dart + ts). MCP-сервер `project` даёт 9 инструментов;
  `/help` перед ответом дёргает `project__git_branch` и `project__git_status` — в ответе видно
  живую ветку, HEAD, тему последнего коммита и грязное ли дерево. Ответ строго JSON
  `{answer, sources, quotes, confidence}`, каждая цитата проверяется кодом на дословное вхождение
  в свой фрагмент — в таблице видно «совпала / не найдена». Вопрос вне базы («как приготовить борщ»)
  отсекается порогом до вызова модели.
- Сверх ТЗ: валидация цитат кодом, честный отказ «нет данных», список источников с близостью,
  MCP отдаёт не только ветку, но и статус, лог, diff, чтение файлов и grep.

### День 32 — Автоматизация ревью кода
- Статус: готово (Action ждёт секрет `PROXYAPI_KEY` и открытия PR — за пользователем)
- Файлы: `week7/devassist/review.py`, `subagents.py`, `.github/workflows/ai-review.yml`,
  демо-ветка `demo/day32-review` (`week7/demo/summarize_journal.py`)
- Задание: пайплайн, где ассистент анализирует новый PR: получает diff и изменённые файлы,
  использует RAG (документация + код), выдаёт текст ревью — потенциальные баги, архитектурные
  проблемы, рекомендации. Пайплайн (например GitHub Action) запускается на PR.
- Что сделано: `.github/workflows/ai-review.yml` на `pull_request` ставит пакет, строит индекс
  `advent` и запускает `python -m devassist.review`, результат постится комментарием в PR через
  `gh pr comment`. Внутри: diff берётся **через MCP** (`project__git_diff` с диапазоном
  `base...head`), по каждому изменённому файлу и по теме измерения делается RAG-поиск,
  дальше **fan-out из трёх субагентов параллельно** — баги / архитектура / конвенции проекта.
  Замечания сортируются по критичности и рендерятся в markdown с блоком «контекст из RAG».
  Демо-ветка содержит скрипт с подсаженными нарушениями; ревью находит: голый `except`,
  off-by-one в подсчёте средней длины, файлы без контекстного менеджера, отсутствие проверки
  ответа API, **захардкоженный ключ со ссылкой на правило 7 CLAUDE.md** и **несуществующую модель
  `gpt-5-turbo` с неверным эндпоинтом** (сверено с тем, как API вызывается в остальном репозитории).
  Прогон: 13 с, три субагента параллельно.
- Сверх ТЗ: fan-out субагентами вместо одного промпта; ревью опирается на писаные правила
  репозитория, а не только на общие практики; тот же код запускается локально командой `/review`.

### День 33 — Ассистент поддержки пользователей
- Статус: готово (ждёт живого прогона на видео)
- Файлы: `week7/devassist/support.py`, `tools_tickets.py`, `servers/tickets_app.py`,
  `week7/data/faq.md`, `week7/data/tickets.json`, `cli.py` (`/support`, `/tickets`, `/demo33`)
- Задание: мини-сервис поддержки: отвечать на вопросы о продукте, использовать RAG (FAQ,
  документация), учитывать контекст пользователя или тикета; через MCP подключить CRM или JSON
  с пользователями/тикетами. Пример: «Почему не работает авторизация?» — ответ с учётом данных тикета.
- Что сделано: второй MCP-сервер `tickets` поверх JSON-CRM (5 пользователей, 6 тикетов с
  перепиской, серверными логами, устройством и версией приложения). Индекс `support` = наш FAQ +
  описание продукта из memory-bank. `/support T-1207` читает тикет через MCP, видит в логах
  `smtp.send FAILED 550` и отвечает пользователю, что письмо не пришло из-за сбоя доставки на нашей
  стороне; `/support T-1211` по логам `pack.grant FAILED` + `wallet.refund NOT EXECUTED` объясняет
  списание монет и **предупреждает не повторять покупку** (правило из FAQ). Вопрос про фичу,
  которой нет («алмазы за турнир») → `confidence: low` → эскалация на инженера, ответ пользователю
  не отправляется. После ответа в тикет пишется заметка (опасная операция — с подтверждением).
- Сверх ТЗ: эскалация по уверенности, заметка в тикет через human-in-the-loop, проверка цитат.

### День 34 — Ассистент для работы с файлами проекта
- Статус: готово (ждёт живого прогона на видео)
- Файлы: `week7/devassist/agentloop.py`, `tools_project.py`, `cli.py` (`/task`, `/sandbox`, `/demo34`)
- Задание: ассистент активно работает с файлами: читает файлы проекта, ищет по нескольким файлам,
  анализирует содержимое, создаёт или изменяет файлы. Минимум 2 сценария; ассистент сам инициирует
  работу с файлами; задача ставится на уровне цели. Изменения сохраняются или выводятся как diff,
  результат воспроизводится повторно.
- Что сделано: LLM-цикл с function-calling поверх Tool Registry — модель сама выбирает инструменты.
  Сценарий 1 (поиск по нескольким файлам): «найди все места, где начисляется или списывается
  игровая валюта» → `rag_search` + `grep` + чтение файлов → таблица файл/строка/что происходит.
  Сценарий 2 (создание файла): «изучи, как устроена авторизация, и создай ADR» → агент сам сходил
  в RAG, сделал grep по `auth|login|token|jwt`, прочитал 5 файлов сервера и записал
  `docs/adr/0001-auth.md`; результат показан как `git diff` песочницы. Запись в живой репозиторий
  отклоняется кодом с объяснением. `/sandbox reset` возвращает песочницу в исходное состояние —
  прогон воспроизводится.
- Сверх ТЗ: защита живого репозитория кодом + подтверждение человека на опасных инструментах,
  таблица шагов агента (какой инструмент, с какими аргументами, чем закончился).

### День 35 — Реальная задача: недельный статус заказчику
- Статус: готово (ждёт живого прогона на видео)
- Файлы: `week7/devassist/status.py`, `cli.py` (`/status`, `/demo35`)
- Задание: придумать реальную задачу, которую действительно хочется автоматизировать; сделать
  приложение/сервис, который использует AI и работает в реальном окружении; описать задачу и
  показать, как AI участвует в процессе.
- Какую задачу решаю: у проекта «Личная культура» есть заказчик (АНО «ИТ-лифт»), которому
  **каждую неделю уходит статус**. Собирать его руками — перечитывать коммиты, задачи и записи
  созвонов. Теперь черновик собирается за 10 секунд.
- Что сделано: `/status 7` через MCP берёт `git_log` за неделю (55 коммитов), читает
  `.memory-bank/tasks/current-tasks.md` и два свежих транскрипта созвонов, а `gpt-4.1` превращает
  это в письмо заказчику: главное за неделю, сделано, в работе, риски и блокеры, вопросы к
  заказчику, план на следующую неделю. Риски вытянуты настоящие (ожидание статического IP под TLS,
  задержка с графдизайнерами, ручная сборка APK). Внизу письма — из каких источников собрано.
  Отправка не автоматическая: черновик показывается человеку и сохраняется в `week7/out/` только
  по подтверждению.
- Сверх ТЗ: метрики по всем вызовам (`/metrics`), retry + fallback на дешёвую модель.

## Что делает пользователь перед видео

1. `pip install -e week7`, переменная `PROXYAPI_KEY` в окружении.
2. `agent31` → `/index` (строит три индекса, ~45 с) → дальше `/demo31` … `/demo35`.
3. Для дня 32 в GitHub: Settings → Secrets and variables → Actions → добавить `PROXYAPI_KEY`,
   затем открыть PR из ветки `demo/day32-review` в `main` — Action прокомментирует PR.
