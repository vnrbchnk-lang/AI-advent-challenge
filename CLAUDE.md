### Harness ≠ API (не путать)
- **Claude Code (harness)** — инструмент, который помогает *писать и запускать код*. Работает на подписке Claude.
- **ProxyAPI/DeepSeek (API из задания)** — облачная LLM, к которой *обращается написанный код* по REST. Именно это требуют задания («облачная модель, доступная через REST»).

### Доступ к ProxyAPI
- Регистрация и ключ: https://console.proxyapi.ru (нужен только email, без иностранного телефона).
- Оплата: рублями с карты любого банка РФ.
- Авторизация во всех запросах: заголовок `Authorization: Bearer <КЛЮЧ>`.
- Базовый адрес: `https://api.proxyapi.ru`, дальше идентификатор провайдера:
  - OpenAI (подтверждено докой): `https://api.proxyapi.ru/openai/v1` → `/chat/completions`
  - DeepSeek: в ProxyAPI **недоступен** (нет в консоли/доке на 2026-06). Не используем.

### Решение: провайдер и модель (зафиксировано 2026-06-03)
- Провайдер: **OpenAI** через ProxyAPI (OpenAI-совместимый формат).
- Эндпоинт: `https://api.proxyapi.ru/openai/v1/chat/completions`.
- Модель: **`gpt-4o-mini`** (подтверждена в списке моделей ProxyAPI: https://proxyapi.ru/docs/openai-models).
- Причина выбора: задания челленджа минимальные → дёшево + быстро + качества хватает, без оверкилла.
- Ключ хранить в переменной окружения (`PROXYAPI_KEY`), НЕ коммитить в код/git.

## 4. ОГРАНИЧЕНИЯ ДЛЯ АГЕНТА (строго соблюдать)

Эти правила обязательны при работе над любым заданием челленджа:

1. **ТЗ должно быть выполнено полностью.** Это учебный проект — полезно и поощряется
   пробовать необычное, выходить за рамки задания (доп. фичи, абстракции, эксперименты),
   если это улучшает понимание материала или просто интересно. Но дословные требования
   задания дня обязаны быть закрыты в любом случае. Когда добавляешь что-то сверх ТЗ —
   явно говори пользователю, что этого не было в формулировке задания.
2. **Не выдумывать.** Если задание чего-то не уточняет — спросить пользователя, а не
   достраивать предположения. Не изобретать несуществующие требования.
3. **Читаемость важнее объёма.** Код может быть сложнее «десятка строк», если это
   оправдано учебной ценностью или фичей. Но держать его простым и понятным, без
   оверинжиниринга ради оверинжиниринга. Сложность добавляем осознанно, а не случайно.
4. **Точность фактов про API.** Не выдумывать эндпоинты, имена моделей, параметры. Если
   не уверен — проверить в актуальной документации или спросить, а не угадывать.
5. **Не выходить за рамки текущего дня.** Не начинать задания будущих дней заранее без
   явной просьбы пользователя.
6. **Соответствие формату сдачи.** Помнить, что результат = Видео + Код. Код должен быть
   таким, чтобы пользователь мог его легко показать и объяснить на видео.
7. **Безопасность.** API-ключи только в переменных окружения, не в коде и не в репозитории.
8. **Сверяться с формулировкой.** Перед сдачей сверять результат с дословной формулировкой
   задания (см. файл `первые2днячелленджа.txt` и последующие).
9. **Журналировать.** После выполнения задания дня дополнять раздел «Журнал по дням» ниже:
   что сделано, какие файлы созданы, ссылка на код, статус сдачи.
10. **Без комментариев в коде.** Код пишется без комментариев (`#` и docstring). Код
    должен быть самодокументирующимся через понятные имена. Пояснения про задачу/выбор
    модели/цены — только в этом CLAUDE.md (журнал по дням), не в `.py`-файлах.
11. **Не запускать сложных агентов самому (экономия токенов).** Агент (Claude Code) НЕ
    гоняет интерактивных/многошаговых агентов и сравнительные прогоны сам — это жжёт
    токены ProxyAPI. Проверка со стороны агента — на уровне **ревью кода** (читаемость,
    логика, корректность вызовов API, верные имена моделей/эндпоинтов). **Запускает живьём
    пользователь** — сразу для видео. Исключение: одиночный быстрый smoke-вызов API, если
    пользователь явно попросил. Дорогие compare-скрипты — пишем, но прогон за пользователем.
12. **Красивый CLI-интерфейс.** Раз делаем CLI — рисовать приятный, читаемый интерфейс в
    стиле Claude Code: рамки/панели, цвета, аккуратные заголовки, таблицы, спиннер при
    запросе. Библиотека — `rich`. Интерфейс не должен мешать сути задания, но выглядеть
    должно симпатично (это идёт на видео).

## 5. Задания (формулировки)

Полные тексты заданий первых двух дней — в файле `первые2днячелленджа.txt`.

### День 1. Первый запрос к LLM через API
Написать минимальный код, который:
- отправляет запрос в LLM через API;
- получает ответ;
- выводит его в консоль или простой интерфейс (CLI / Web).
**Результат:** код, отправляющий запрос в LLM через API и получающий ответ.
**Формат:** Видео + Код.

### День 2. Формат ответа
Отправить тот же запрос, но:
- добавить явное описание формата ответа;
- добавить ограничение на длину ответа;
- добавить условие завершения ответа (stop sequence или явная инструкция).
Сравнить ответы: без ограничений vs с ограничениями.
**Результат:** один и тот же запрос с разным уровнем контроля ответа через API.
**Формат:** Видео + Код.

### День 3. Разные способы рассуждения
Взять одну задачу (логическую/алгоритмическую/аналитическую) и решить её через API
четырьмя способами:
- прямой ответ без доп. инструкций;
- инструкция «решай пошагово»;
- модель сначала сама составляет промпт для задачи, затем решает по нему;
- группа экспертов в промпте (аналитик, инженер, критик) — решение от каждого.
Сравнить: отличаются ли ответы, какой способ точнее.
**Результат:** несколько решений одной задачи и их сравнение.
**Формат:** Видео + Код.

### День 4. Температура
Выполнить один и тот же запрос с `temperature` = 0 / 0.7 / 1.2. Сравнить ответы по
точности, креативности, разнообразию. Сформулировать, для каких задач лучше каждая настройка.
**Результат:** примеры ответов с разной температурой и выводы.
**Формат:** Видео + Код.

## 7. Неделя 3 — память, состояние, stateful-агент (текущая)

> Контекст недели — конспект `03-memory-state-summary.md`. **Всегда сверяться с ним по теории.**
> Темы: stateless→stateful, персонализация (профиль), memory layers, task state machine,
> инварианты, prompt builder, validation, save/resume. Цель недели — собрать всё это в
> один **stateful-агент** к концу недели.

### Архитектурное решение недели 3 (зафиксировано 2026-06-18)
- **Единый растущий пакет, а не 5 отдельных скриптов.** Папка `week3/assistant/` — один
  пакет, каждый день добавляет одну абстракцию. К концу недели = собранный stateful-агент
  (это и есть цель недели по конспекту: «собрать части в управляемую систему»).
- Планируемая раскладка (уточняется по ходу заданий, будущие дни заранее НЕ пишем):
  - `memory.py` — день 11 (memory layers): short-term / working / long-term.
  - `profile.py` — персонализация (стиль / constraints / контекст).
  - `state.py` — task state machine (стадии + allowed transitions). **Сюда же — переключение между задачами** (именованные таски, отдельная рабочая память на задачу): отложено из дня 11 осознанно, ляжет системно в state machine. В дне 11 рабочая память — один плоский KV-блок без мультизадачности (метод `clear_working()` в `memory.py` есть, в CLI не выведен).
  - `invariants.py` — инварианты + проверка кодом (линтер требований).
  - `prompt_builder.py` — сборка prompt из выбранных слоёв.
  - `agent.py` — собранный stateful-агент (LLM engine + всё выше).
  - `demoNN.py` — демо/сравнение для каждого дня (прогон за пользователем — см. правило 11).
- Хранилище раздельное по слоям: каталог `week3/assistant/store/` с отдельными JSON на
  слой (видно «хранится отдельно» — для видео нагляднее одного файла).
- Модель: **`gpt-5.2`** (как неделя 2), дешёвый экстрактор — **`gpt-4o-mini`**. Эндпоинт и
  формат — те же (ProxyAPI OpenAI-совместимый). Прайс — ProxyAPI (https://proxyapi.ru/pricing/list).
- CLI — красивый, через `rich` (правило 12).

### Задание дня 11 (дословно)
Описать и реализовать модель памяти ассистента. Разделить информацию минимум на 3 типа:
краткосрочная (текущий диалог), рабочая (данные текущей задачи), долговременная (профиль,
решения, знания). Сделать: разные типы памяти хранятся отдельно; явно выбирается, что и
куда сохраняется. Проверить: какие данные попадают в каждый слой; как это влияет на ответы.
**Результат:** агент с явной моделью памяти (memory layers). **Формат:** Видео + Код / Текст.
