6. **Формат сдачи.** Результат = Видео + Код. Код должен быть легко показуем и объясним на видео.
7. **Безопасность.** API-ключи только в переменных окружения, не в коде и не в репозитории.
8. **Сверяться с формулировкой.** Перед сдачей сверять с дословным ТЗ (см. `первые2днячелленджа.txt` и последующие).

Проверка агента — **ревью кода** (читаемость, логика, корректность вызовов API, верные имена моделей/эндпоинтов). **Запускает живьём пользователь** — сразу для видео. Исключение: одиночный быстрый smoke-вызов API, если пользователь явно попросил. Дорогие compare-скрипты — пишем, прогон за пользователем.
12. **Красивый CLI.** Рисовать приятный, читаемый интерфейс в стиле Claude Code: рамки/панели, цвета, аккуратные заголовки, таблицы, спиннер при запросе. Библиотека — `rich`. Интерфейс не мешает сути, но выглядит симпатично (идёт на видео).

## 5. Задания (формулировки)

Полные тексты первых двух дней — в `первые2днячелленджа.txt`.

### День 1. Первый запрос к LLM через API
Минимальный код, который:
- отправляет запрос в LLM через API;
- получает ответ;
- выводит в консоль или простой интерфейс (CLI / Web).
**Результат:** код, шлющий запрос в LLM через API и получающий ответ.
**Формат:** Видео + Код.

### День 2. Формат ответа
Тот же запрос, но:
- добавить явное описание формата ответа;
- добавить ограничение длины;
- добавить условие завершения (stop sequence или явная инструкция).
Сравнить: без ограничений vs с ограничениями.
**Результат:** один запрос с разным уровнем контроля ответа через API.
**Формат:** Видео + Код.

### День 3. Разные способы рассуждения
Одна задача (логическая/алгоритмическая/аналитическая), решить через API четырьмя способами:
- прямой ответ без доп. инструкций;
- инструкция «решай пошагово»;
- модель сама составляет промпт для задачи, затем решает по нему;
- группа экспертов в промпте (аналитик, инженер, критик) — решение от каждого.
Сравнить: отличаются ли ответы, какой способ точнее.
**Результат:** несколько решений одной задачи + сравнение.
**Формат:** Видео + Код.

### День 4. Температура
Один запрос с `temperature` = 0 / 0.7 / 1.2. Сравнить по точности, креативности, разнообразию. Сформулировать, для каких задач лучше каждая настройка.
**Результат:** примеры ответов с разной температурой + выводы.
**Формат:** Видео + Код.

### День 15. Контролируемые переходы состояний
Явные переходы между состояниями задачи: допустимые состояния; разрешённые переходы; ассистент не «перепрыгивает» этап (нет реализации до утверждённого плана; нет финала без валидации). Проверить: попытки перейти в недопустимое состояние, реакцию ассистента, корректность продолжения после паузы.
**Результат:** ассистент с контролируемым жизненным циклом задачи. **Формат:** Видео + Код / Текст.

## 7. Неделя 3 — память, состояние, stateful-агент (текущая)

> Контекст недели — конспект `03-memory-state-summary.md`. **Всегда сверяться по теории.**
> Темы: stateless→stateful, персонализация (профиль), memory layers, task state machine,
> инварианты, prompt builder, validation, save/resume. Цель недели — собрать всё в один **stateful-агент** к концу недели.

### Архитектурное решение недели 3 (зафиксировано 2026-06-18)
- **Единый растущий пакет, не 5 отдельных скриптов.** `week3/assistant/` — один пакет, каждый день добавляет одну абстракцию. К концу недели = собранный stateful-агент (цель недели по конспекту: «собрать части в управляемую систему»).
- Планируемая раскладка (уточняется по ходу, будущие дни заранее НЕ пишем):
  - `memory.py` — день 11 (memory layers): short-term / working / long-term.
  - `profile.py` — персонализация (стиль / constraints / контекст).
  - `state.py` — task state machine: `TaskStore` — **мультизадачный** (словарь именованных задач + указатель `current`), стадии + allowed transitions, статусы active/paused/done. **Переключение между задачами реализовано** (просьба пользователя): `/task <имя>` открывает рабочее пространство задачи (вложенный prompt-loop, стадии `/next`/`/back`, `/pause`, `/delete`); список задач со статусами всегда в нижней панели CLI (`bottom_toolbar`).
  - `invariants.py` — инварианты + проверка кодом (линтер требований).
  - `swarm.py` — день 15 (рой агентов): на **каждой** рабочей стадии свой рой из 5 ролей
    (`gpt-4o-mini`) + оркестратор (`gpt-4.1`) сводит мнения и возражает при конфликте с инвариантами.
    Роли по стадиям: planning (Архитектор/Безопасность/Производительность/Прагматик/Критик),
    execution (Бэкенд/Данные/Тесты/Интеграции/Критик кода), validation (Инварианты/Безопасность/
    Граничные случаи/Производительность/Критик); done — без роя, финальная сводка.
    Рой стартует **авто при входе на стадию** (создание задачи → planning; каждый `/next` →
    рой новой стадии). Итог planning = утверждённый план (связь со state machine: без
    плана нет execution); итоги остальных стадий пишутся в результаты задачи. **Стадию двигает
    только человек** командой `/next`/`/back` — LLM перехода не имеет (ядро дня 15). `/council` —
    ручной перезапуск роя текущей стадии.
  - `prompt_builder.py` — сборка prompt из выбранных слоёв.
  - `agent.py` — собранный stateful-агент (LLM engine + всё выше).
  - демо/сравнение каждого дня — командой в CLI (день 11 — `/demo`) либо отдельным скриптом; прогон за пользователем (правило 11).
- Хранилище раздельное по слоям: каталог `week3/assistant/store/` с отдельным JSON на слой (видно «хранится отдельно» — для видео нагляднее одного файла).
- Модель: **`gpt-4.1`** (сменили с gpt-5.2 ради экономии — выход 2062₽/1М против 4245₽; цена сверена на https://proxyapi.ru/pricing/list). Дешёвый экстрактор для будущих дней (извлечение профиля/фактов), если надо, — **`gpt-4o-mini`**. Эндпоинт и формат — те же (ProxyAPI OpenAI-совместимый).
- CLI — красивый, через `rich` + `prompt_toolkit` (слэш-команды с автодополнением); основной акцентный цвет — **тёмно-синий** (navy `#34568b`, константы `NAVY*`; сменён со стального серого на день 15 по просьбе пользователя), без эмодзи (правило 12).

Проверить: попытки недопустимого перехода, реакцию, продолжение после паузы. Реализация: ядро (`TRANSITIONS`, код-гейт `transition()`, плательный гейт execution, done терминальна) стояло с дня 13 — день 15 добавил демо `/demo5` (детерминированно показывает отказ кода на нелегальных переходах + пауза/резюм) и **рой агентов** `/council` (`swarm.py`, сверх дословного ТЗ — по просьбе оргкомитета: 5 ролей + оркестратор-арбитр). **Результат:** ассистент с контролируемым жизненным циклом задачи.
