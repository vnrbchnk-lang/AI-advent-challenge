## 5. Задания (формулировки)

Полные тексты заданий первых двух дней — в файле `первые2днячелленджа.txt`.

### День 1. Первый запрос к LLM через API
Написать минимальный код, который:
- отправляет запрос в LLM через API;
- получает ответ;
- выводит его в консоль или простой интерфейс (CLI / Web).
**Результат:** код, отправляющий запрос в LLM через API и получающий ответ.
**Формат:** Видео + Код.

### День 2. Формат ответа
Отправить тот же запрос, но:
- добавить явное описание формата ответа;
- добавить ограничение на длину ответа;
- добавить условие завершения ответа (stop sequence или явная инструкция).
Сравнить ответы: без ограничений vs с ограничениями.
**Результат:** один и тот же запрос с разным уровнем контроля ответа через API.
**Формат:** Видео + Код.

### День 3. Разные способы рассуждения
Взять одну задачу (логическую/алгоритмическую/аналитическую) и решить её через API
четырьмя способами:
- прямой ответ без доп. инструкций;
- инструкция «решай пошагово»;
- модель сначала сама составляет промпт для задачи, затем решает по нему;
- группа экспертов в промпте (аналитик, инженер, критик) — решение от каждого.
Сравнить: отличаются ли ответы, какой способ точнее.
**Результат:** несколько решений одной задачи и их сравнение.
**Формат:** Видео + Код.

### День 4. Температура
Выполнить один и тот же запрос с `temperature` = 0 / 0.7 / 1.2. Сравнить ответы по
точности, креативности, разнообразию. Сформулировать, для каких задач лучше каждая настройка.
**Результат:** примеры ответов с разной температурой и выводы.
**Формат:** Видео + Код.

### День 5 — Версии моделей
- Статус: готово (код работает, разница моделей видна)
- Файлы: `week1/day5/day5.py`
- Запрос (ловушка на внимательность): «У Салли 3 брата. У каждого из её братьев есть 2 сестры. Сколько сестёр у Салли?» — в семье 2 сестры (Салли + ещё одна), поэтому у каждого брата ровно 2 сестры, а у самой Салли сестёр = 1. Слабая модель тупо умножает/складывает; сильная рассуждает.
- Что сделано: один `ask(model, prompt)`-хелпер. Один запрос × 3 модели (слабая/средняя/сильная): `gpt-3.5-turbo` / `gpt-4o` / `gpt-5.5`. Замеряет **время** (`time.perf_counter`), **токены** (`usage`: вход/выход/всего). Печатает по каждой модели: время, токены, цена/1М (вход/выход), ответ. Цену за конкретный запрос не считаем — её пользователь смотрит на сайте ProxyAPI.
- Токены: **вход** (prompt) = отправленные модели сообщения (платишь за чтение); **выход** (completion) = сгенерированный ответ (платишь за генерацию, обычно дороже входа в разы).
- Цены (за 1М токенов, всё в USD единой валютой): gpt-3.5-turbo $0.50/$1.50, gpt-4o $2.50/$10.00 — официальный прайс OpenAI; gpt-5.5 — новее cutoff, цены OpenAI нет → прайс ProxyAPI 1520₽/9100₽, конвертирован в $ через константу `RUB_PER_USD = 90` (поправить при другом курсе).
- Замечания по API: `-pro`-модели (o1-pro, gpt-5-pro, gpt-5.5-pro) в `/chat/completions` дают **404** — они только на `/responses`, поэтому сильная = `gpt-5.5` (не pro). Имена моделей сверены с живым списком `GET /openai/v1/models`.
- Результат сравнения (фактический прогон): 3.5-turbo — заученная многоходовка, подвох не заметила (❌), ~2.0 c, 270 токенов; gpt-4o — заметила подвох, «один рейс» (✅), ~1.4 c, 155 токенов; gpt-5.5 — заметила подвох, ответ чище всех (✅), ~3.4 c, 143 токена. Вывод: сильнее модель → читает условие внимательнее (качество), тратит меньше токенов на суть, но медленнее и дороже за токен.

### День 8 — Работа с токенами
- Статус: готово (проверено живыми запросами: подсчёт, симуляция переполнения, реальная ошибка API)
- Файлы: `week2/day8/agent8.py`, `week2/day8/pyproject.toml`
- Задание: подсчёт токенов (текущий запрос / вся история / ответ); сравнить короткий, длинный и переполняющий лимит диалоги; показать рост стоимости и что ломается при переполнении.
- Что сделано (на базе day7 `Agent`): все три числа — из `usage` ответа API, без `tiktoken`. `prompt_tokens` = вся отправленная история (модель каждый ход читает её целиком), `completion_tokens` = ответ, «текущий запрос» = дельта: `prompt_tokens` минус `history_tokens` прошлого хода (проверено: ход2 запрос 12 = 205−193 ✅). Стоимость в ₽ по прайсу ProxyAPI (https://proxyapi.ru/pricing/list): gpt-5.2 — 531₽ вход / 4245₽ выход за 1М. После каждого хода печатается строка: запрос | вход всего | ответ | история N/лимит (%) | цена хода | цена всего.
- Переполнение — два механизма (решение пользователя: ручной лимит; реальный flood добавлен как страховка по дословному ТЗ «лимит модели»):
  1. **Ручной лимит** `CONTEXT_LIMIT = 10000` токенов на gpt-5.2 (настоящее окно 400k — честно переполнять долго/дорого: ~207₽ за ход при 390k входа). История ≥ лимита → `ContextOverflowError`: запрос к модели НЕ отправляется, агент отказывается отвечать до `reset`. Предупреждение при ≥80%. Симуляция поведения реального API. После переполнения модель не отвечает НИКАК — ошибка вместо ответа и есть «что ломается», у реального API так же (400 на валидации, модель запрос не видит).
  2. **Команда `flood`** — реальное переполнение: одно сообщение «вода »×20000 (~40k токенов) в gpt-3.5-turbo (окно 16 385) → настоящий HTTP 400: `This model's maximum context length is 16385 tokens. However, your messages resulted in 40008 tokens.` Запрос отклонён валидацией до тарификации. История агента не затрагивается (отдельный одноразовый запрос).
- **Команда `fill N`** (идея пользователя): дописывает в историю ~N токенов балласта **локально, без запроса к API** (бесплатно) + ответ «Принято, игнорирую». Платным становится следующий вопрос — он несёт всю раздутую историю. Демо «влияние заполнения»: `fill 7000` → история ~70% лимита → обычный вопрос стоит ~4₽ вместо 0.2₽ (скачок цены хода в ~20 раз виден в счётчике), дальше предупреждение 80% и перелив за пару реплик. Оценка токенов: «вода » = **1 токен у gpt-5.2** (o200k-токенизатор, кириллица плотнее), но **2 токена у gpt-3.5-turbo** (flood: 20000 повторов = 40008 ток.) — токенизаторы у моделей разные! Проверено: fill 1000 → реальный вход 1083 (дрейф ~5% — префикс балласта + вопрос); первая версия с коэффициентом 2 давала вход 3583 вместо 7000 и отрицательную дельту запроса.
- Формат `history.json` расширен: `{"messages": [...], "history_tokens": N, "turns": [...]}` — счётчик токенов и журнал ходов переживают перезапуск (ручной лимит работает и после рестарта). Команда `stats` — таблица по ходам: видно рост «вход всего» с каждой репликой (161 → 205 → ...) и накопление цены.
- Глобальная команда: **`agent8`** (`pip install -e week2\day8`), модуль `agent8.py` (не `agent.py` — коллизия top-level, урок day7).
- Фикс кодировки: piped-stdin на Windows давал суррогаты (`UnicodeEncodeError: 'utf-8' codec can't encode character '\udc98'... surrogates not allowed` при сохранении JSON) → добавлен `sys.stdin.reconfigure(encoding="utf-8", errors="replace")` рядом со stdout-reconfigure. После фикса pipe-тест чистый (без BOM-артефактов day7), интерактивный терминал работает как раньше.
- Сверх ТЗ: команды `stats` (таблица ходов) и `fill N`, персистентный счётчик токенов/стоимости между запусками, предупреждение при ≥80% заполнения.
- Сценарий демо для видео: короткий диалог (пара реплик, единицы % лимита, ~0.2₽/ход) → `fill 7000` (история ~70%) → тот же вопрос стоит ~4₽ (рост цены наглядно) → ещё реплики → предупреждение 80% → перелив ручного лимита (агент отказывает, «что ломается») → `flood` (настоящая ошибка API на реальном окне модели) → `stats` (таблица роста) → `reset` (агент ожил).

### День 10 — Управление контекстом: разные стратегии (без summary)
- Статус: готово (проверено живым прогоном compare10.py — 3 стратегии, 56 вызовов, чёткий контраст)
- Файлы: `week2/day10/agent10.py`, `week2/day10/compare10.py`, `week2/day10/pyproject.toml`
- Задание: реализовать 3 стратегии управления контекстом + переключатель: (1) Sliding Window — только последние N сообщений, остальное отбрасывать; (2) Sticky Facts / KV-память — отдельный блок facts (ключ-значение), обновлять после каждого сообщения юзера, слать facts + последние N; (3) Branching — checkpoint, 2 ветки от одной точки, продолжать независимо, переключаться. Прогнать один ТЗ-сценарий через каждую, сравнить качество/стабильность/токены/удобство. **Явно: без summary** (это было day9).
- Архитектура (на базе day8/9): единый класс `Agent`, поле `self.strategy` ∈ {sliding, facts, branching}, переключатель CLI `strategy <имя>`. Стратегия влияет ТОЛЬКО на `build_request_messages()` — что из хранения уходит в API. Хранилище универсальное: `branches` (dict имя→список сообщений) + `current_branch`; в sliding/facts работаем в ветке `main`. `self.messages` — property на `branches[current_branch]`. System-промпт хранится отдельно, не в истории (как day9). Искусственный лимит/overflow day8 убран — не тема дня.
- Стратегия 1 (sliding): `build_request_messages` = `[system] + messages[-window_n:]`. Всё старше окна не уходит в запрос.
- Стратегия 2 (facts): после реплики юзера `update_facts()` зовёт ДЕШЁВУЮ модель **gpt-4o-mini** (39₽/155₽ за 1М — в ~14× дешевле gpt-5.2 на входе), та возвращает обновлённый JSON KV (цель/ограничения/решения/числа/сроки). Парсинг с защитой от ```-обёрток и не-dict. В запрос: `[system] + [system: facts JSON] + messages[-window_n:]`. Facts обновляются ДО ответа → текущий ход уже видит свежий KV. Выбор дешёвой модели — решение пользователя («слабее и дешевле, но чтобы справлялась»). Подтверждённый прайс ProxyAPI (https://proxyapi.ru/pricing/list).
- Стратегия 3 (branching): `make_checkpoint()` = deepcopy текущих сообщений в `self.checkpoint`; `branch(name)` = новая ветка из checkpoint (или текущей точки), переключение на неё; `switch(name)` — между ветками. «2 ветки от одного места»: checkpoint держится до перезаписи, два `branch A`/`branch B` форкаются из одной точки и расходятся. В запрос branching шлёт ПОЛНУЮ текущую ветку (стратегия не про обрезку, а про параллельные истории).
- Учёт токенов day8 расширен на 2 модели: `cost_rub(model, …)`, `turns` хранят `kind` (ask/facts), `model`, `strategy`, `branch`. `stats` — таблица с разбивкой. Персистенция (решение пользователя — сохранять): `history.json` = strategy/window/branches/current/checkpoint/facts/turns/tokens, всё переживает перезапуск.
- Команды CLI: `strategy sliding|facts|branching`, `window N`, `facts` (показать KV), `checkpoint`, `branch <имя>`, `switch <имя>`, `branches` (список + текущая), `stats`, `reset`, `exit`. Промпт строки показывает `[стратегия/ветка]`.
- Сравнение — `compare10.py` (решение пользователя — скрипт + живой CLI): ТЗ-сценарий «собираем ТЗ на приложение доставки» 12 реплик (важные факты — бюджет 500к, срок 1 сентября, платформа Android, цвет #2E7D32, оплата ЮKassa — в ПЕРВЫХ сообщениях, окно N=4 → они выпадают), затем 2 контрольных вопроса на ранние детали. Прогон через 3 стратегии (свой history-файл, чистится сам).
